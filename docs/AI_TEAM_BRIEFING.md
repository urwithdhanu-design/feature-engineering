# AI Team Briefing — Credit Card Nudge Platform

A short guide for explaining this project in meetings with the core AI / ML team.

---

## 1. Elevator pitch (30 seconds)

> We built a **Python ML pipeline** that takes a credit card customer’s **payment history and account state**, and returns a **personalised nudge** for the UI: which message type to show, what tone to use, payment risk scores, and £20 reward eligibility.  
> It uses **feature engineering**, **data cleaning**, **K-Means segmentation**, and **gradient boosting models** — not an LLM. Messages come from **approved templates**.

---

## 2. What problem does it solve?

| Business need | Platform output |
|---------------|-----------------|
| Reduce missed payments | `warning` / `motivational` nudges |
| Promote direct debit | `direct_debit_setup` nudge |
| Drive £20 reward uptake | `reward_led` nudge + eligibility rules |
| Right message for age group | Tone: professional / casual / supportive |
| Prioritise risky accounts | Payment risk probabilities + `high_risk` segment |

---

## 3. End-to-end architecture

```
input/customer.json
        ↓
Data cleaning (validate, dedupe, fix values)
        ↓
Feature engineering (payment stats, utilization, cycle codes)
        ↓
K-Means → behavioural segment (reliable / at_risk / high_risk)
        ↓
ML models → nudge type, tone, risk scores
        ↓
Business rules → reward eligibility, overrides
        ↓
Message templates → final UI text
        ↓
output/customer_recommendation.json
```

## 4. What does “training a model” mean?

**Training** = showing an algorithm many **examples** so it learns patterns to **predict outcomes for new customers**.

### Analogy

Like teaching someone to sort mail:
1. You give them **5,000 labelled letters** (training data)
2. They learn: *“This shape of address usually means urgent”*
3. Tomorrow, a **new letter** arrives — they classify it **without** having seen that exact letter before

### In this project

| Step | What happens |
|------|----------------|
| **Training data** | 5,000 synthetic customers (`synthetic_5000_2mo.jsonl`) |
| **Features (inputs)** | 17 numbers per customer (age, utilization, missed payments, segment, …) |
| **Labels (targets)** | What nudge to show, tone, risk scores (from rules + heuristics) |
| **Training** | Algorithm adjusts internal rules to minimise prediction error |
| **Saved artifact** | `models/model_bundle.pkl` — reused at inference |
| **Inference** | New customer JSON → same pipeline → prediction (no retraining) |

**Important:** Training is **offline** (batch). Inference is **online** (per customer, milliseconds).

## 7. Training data & model architecture (detailed)

Use this section when the AI team asks *“What exactly are you training on?”* and *“What is the base model?”*

### 7.1 Training data source

| Property | Value |
|----------|--------|
| **Default file** | `data/training/synthetic_5000_2mo.jsonl` |
| **Records** | 5,000 synthetic customers |
| **History per customer** | 2 months (`payment_history_string`, e.g. `"MX"`, `"MM"`) |
| **Real bank data?** | **No** — generated by `python main.py generate-synthetic` |
| **Fallback** | 177 persona matrix JSONs in `data/scenarios/` (if synthetic file missing) |
| **Train / test split** | 4,000 train / 1,000 test (80/20, `random_state=42`) |

Metrics are saved in `models/training_metrics.json` after each train run.

### 7.2 What each training row contains

Each line in the JSONL file is one **fake credit card account**:

| Field group | Examples |
|-------------|----------|
| Identity | `customerId`, `productId`, `age` (18–55) |
| Payment behaviour | `payment_history_string` (2 chars: M/D/L/X/N) |
| Balances | `current_balance_amount`, `account_credit_limit_amount`, utilization |
| Direct debit | `direct_debit_active_indicator`, `direct_debit_type` |
| Derived (at train time) | missing payments, arrears, on-time counts |

**Payment history generation weights** (`scripts/generate_synthetic_training_data.py`):

| Code | Weight | Meaning |
|------|--------|---------|
| M | 40% | Paid on time |
| D | 10% | Paid early |
| L | 15% | Paid late |
| X | 20% | Missed payment |
| N | 15% | Never paid |

### 7.3 What the ML models actually see (inputs)

After **cleaning → feature engineering → K-Means**, each row becomes **17 numeric features**:

| # | Feature | Description |
|---|---------|-------------|
| 1 | `age` | Customer age |
| 2 | `months_on_book_counter` | Months since account opened |
| 3 | `utilization` | balance ÷ credit limit |
| 4–6 | `payment_*_due_amount` | Total, current, past-due amounts |
| 7–8 | `missing_payments_last_6_cycles`, `on_time_payments_last_6_cycles` | From payment history |
| 9–10 | `_late_payments`, `_never_paid_cycles` | Derived counters |
| 11 | `direct_debit_active_indicator` | 0 or 1 |
| 12 | `reward_qualifying_on_time_count` | On-time M payments for reward |
| 13 | `has_arrears` | 0 or 1 |
| 14–16 | `cycle_1/2/3_code` | M=0, D=1, L=2, X=3, N=4, no data=5 |
| 17 | `customer_segment` | K-Means cluster id (0, 1, 2) |

Full list: `FEATURE_COLUMNS` in `src/features/engineer.py` + `customer_segment`.

### 7.4 What the models learn to predict (labels)

Labels are **not** from real historical outcomes. They are **rule-generated** at training time:

| Target | Source | File |
|--------|--------|------|
| **Nudge type** | Business rules | `src/rules/nudge_rules.py` → `derive_nudge_type()` |
| **Tone** | Age / persona mapping | `derive_tone()` |
| **Risk scores** | Heuristics from payment history | `derive_risk_labels()` |

The models learn to **reproduce rule behaviour** from features — useful as a POC; production should use **real outcome labels** (who actually paid late, who responded to nudges).

### 7.5 Training pipeline (step by step)

```
data/training/synthetic_5000_2mo.jsonl   (5000 account JSON lines)
        ↓
1. Data cleaning          src/data/cleaning.py
        ↓
2. Feature engineering  src/features/engineer.py  (16 features)
        ↓
3. K-Means (k=3)        src/data/segmentation.py  (+customer_segment = 17th)
        ↓
4. Rule-based labels    nudge, tone, risk
        ↓
5. train_test_split     4000 / 1000
        ↓
6. sklearn GradientBoosting fit
        ↓
models/model_bundle.pkl + segment_profiles.json + training_metrics.json
```

**Inference** (`POST /v1/recommend`) runs steps 1–2 and 6 only — **never retrains**.

### 7.6 What is the “base model”?

Two meanings — both matter in meetings:

#### A) Pre-trained / foundation model (GPT-style)? → **None**

| Question | Answer |
|----------|--------|
| Pre-trained LLM? | **No** |
| Transfer learning? | **No** |
| Weights from external model? | **No** |

Models are trained **from scratch** on synthetic tabular data each time you run `python main.py train`.

> **Meeting line:** *“We don’t fine-tune a foundation model. We train small supervised sklearn models from scratch on engineered features.”*

#### B) Base learner inside Gradient Boosting? → **Shallow decision tree (CART)**

sklearn’s `GradientBoostingClassifier` / `GradientBoostingRegressor` use **decision trees** as weak learners.

| Model | sklearn class | `n_estimators` | `max_depth` | Task |
|-------|---------------|----------------|-------------|------|
| Nudge | `GradientBoostingClassifier` | 100 | 4 | Classification |
| Tone | `GradientBoostingClassifier` | 100 | 3 | Classification |
| Risk ×3 | `GradientBoostingRegressor` | 80 | 3 | Regression |

**Not used:** XGBoost, LightGBM, CatBoost, TensorFlow, PyTorch.

**How boosting works:**

```
Tree 1  → initial prediction on 17 features
Tree 2  → corrects Tree 1’s errors
Tree 3  → corrects remaining errors
   ...
Tree N  → final ensemble prediction
```

Each tree is **shallow** (`max_depth` 3–4) to avoid overfitting on 5,000 rows.

Implementation: `src/models/train.py`  
Saved artifact: `models/model_bundle.pkl` (pickle — contains all 5 models + K-Means segmenter + encoders).

## 10. K-Means clusters (behavioural, not age)

**3 clusters by default** — named by **payment risk** after fitting:

| Name | Meaning |
|------|---------|
| `reliable` | Low risk — on-time payers, no arrears |
| `at_risk` | Medium risk — some misses/late, may have arrears |

## 14. Likely questions from core AI team

**Q: What is the base model?**  
A: No foundation model. sklearn Gradient Boosting with shallow decision trees as weak learners (`max_depth` 3–4, 80–100 trees). Trained from scratch on 5,000 synthetic rows.

**Q: Why not XGBoost / LightGBM?**  
A: sklearn is sufficient for this POC; same GBDT family. Easy upgrade path later on real data.

**Q: Why not an LLM?**  
A: Banking needs predictable, auditable messages. Templates + classical ML are faster, cheaper, and compliant.

**Q: Why gradient boosting vs deep learning?**  
A: Small tabular dataset, interpretable features, no need for neural nets on this scale.

**Q: Why K-Means with k=3?**  
A: Default for 3 risk tiers; configurable via `--clusters`. Names assigned by risk score post-hoc.

**Q: Data leakage?**  
A: Train/test split on customers; segmenter fit on full set before split — in production, fit segmenter on train only only (improvement for v2).

**Q: How do you evaluate?**  
A: Holdout accuracy/MAE today; production needs A/B tests on payment outcomes.

**Q: MLOps / deployment?**  
A: Today: CLI + pickle file. Production: wrap `recommend_for_account()` in an API, version models, monitor drift.
